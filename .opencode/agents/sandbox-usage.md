# OpenCHAMI Integration Sandbox – Usage Guide

## Primary entry point

```bash
make ci
```

Runs the full lifecycle:  
1️⃣ `make build-images` – builds any locally-required images (`power-control`,
   `ipmi_sim`).  
2️⃣ `make up` – starts the Docker-Compose stack.  
3️⃣ `make seed` – loads Vault, S3 buckets, PostgreSQL, and SMD component data.  
4️⃣ `make test` – executes the Go integration suite (`make test`) **and** the
   BATS smoke tests.  
5️⃣ `make down` – shuts down everything and removes containers.

## Useful shortcuts

| Target | What it does |
|--------|--------------|
| `make build-images` | Build only the two services that are not pulled from GHCR. |
| `make up` | Start the stack without seeding or testing (good for manual debugging). |
| `make seed` | Run only the fixture-loading scripts (`scripts/*.sh`). |
| `make test` | Execute the Go integration tests **or** the BATS suite (`make test-bats`). |
| `make down` | Stop containers; `make down -v` also removes volumes (use only inside the script that performed `up`). |
| `make tail` | Tail `PROGRESS.log` and `STATUS` for a phone-friendly view of progress. |
| `make show-images` | Print the resolved image manifest (`images/*.env`). |
| `make refresh-releases` | Regenerate the `images/*.env` files, pinning each service to its latest GitHub Release tag. |
| `make ci DEBUG=1` | Run with `set -x`-style debugging output (useful for CI-log analysis). |

## Overriding a service image

Environment variables prefixed with `SBX_` allow per-service overrides:

```bash
SBX_SMD_IMAGE=ghcr.io/openchami/smd:pr-123 \
SBX_TOKENSITH_IMAGE=ghcr.io/openchami/tokensmith:pr-456 \
make ci
```

The CI integration docs (`docs/ci-integration.md`) detail how service
repositories embed this pattern in their own GitHub Actions.

## Refreshing the release manifest

```bash
make refresh-releases   # pulls latest tags from GitHub and writes images/release.env
```

The default `make ci` uses `IMAGES=release` (the `release.env` file).  
Switch to floating tags with `IMAGES=default` or to the `main` branch builds
with `IMAGES=edge`.

## Troubleshooting hints (quick reference)

| Symptom | Likely cause | First-step fix |
|---------|--------------|----------------|
| A container stays in `starting` / `unhealthy` | Missing fixture data or vault token mismatch | Run `make down && make up` to reset; verify `PROGRESS.log` for the failure step. |
| `make ci` aborts after "waiting for stack" | One of the infra services (Vault, LocalStack, Postgres) failed to become healthy | Inspect `logs/<UTC>/` for the collected logs; run `scripts/log-bundle.sh` manually to gather more detail. |
| BMC-simulator cannot be reached | Host-name alias not resolved | Ensure Docker network `openchami-sandbox` exists (created by `make up`) and that `/etc/hosts` inside the containers contains the `x0c0s…` entries (generated by `compose/bmc-sim.yaml`). |
| Tests report "token verification failed" (UC4) | JWKS URL not reachable or token-signing key mismatch | Verify that the `tokensmith` container logs show a successful bootstrap token creation; check `SBX_TOKENSITH_IMAGE` if you overrode the image. |
| FRU-tracker snapshot not reconciled (UC7) | `inventory-snapshot.json` fixture not loaded or SQLite DB not initialized | Re-run `make seed`; ensure `fixtures/seed-smd.sh` completed without errors. |

All failures are captured in a **log bundle** (`scripts/log-bundle.sh`) that
writes a timestamped directory under `logs/`. The bundle contains:

* `docker compose ps` output  
* `docker ps -a` output  
* `docker compose logs --tail=200 <service>` for the failing service  

The bundle path is echoed to `PROGRESS.log` for easy retrieval.

## Extending the sandbox

When adding a new service, fixture, or test:

1. Add the component under the appropriate directory (`compose/`, `fixtures/`,
   `tests/integration/`).  
2. Update the reference cards in `docs/` (e.g., `reference-endpoints.md`).  
3. **Do not** edit any external repository – file an upstream issue instead.  
4. Follow the conventions listed in `.claude/CLAUDE.md` (no new ports,
   credentials, or xnames without updating the docs).

The agent can answer "How do I add a new service?" or "Where should I place a new fixture?" with step-by-step guidance that respects the sandbox's hard rules.
